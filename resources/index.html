Rentpath Property and Reviews REST API Resources

This is the reference document for the Rentpath Property and Reviews REST API. The REST API is for developers who want to:

  • integrate reviews with other applications
  • create scripts that interact with reviews

Getting started

Because the REST API is based on open standards, you can use any web development language or command line tool capable of generating an HTTP request to access the API.

Structure of the REST URIs

Rentpath's REST APIs provide access to resources (data entities) via URI paths To use a REST API, your application will make an HTTP request and parse the response. The Rentpath REST API uses JSON as its communication format, and the standard HTTP methods like GET, PUT, POST and DELETE. URIs for Rentpath's REST API resource have the following structure: 

    http://host:port/context/version/path/to/resource?p1=v1&p2=v2...

For example, the following URI would retrieve reviews for January 1, 2013 for property 999. 

    https://api.reviews.apartmentguide.com/v1/reviews?property-id=999&date=01012013

See the API descriptions below for a full list of available resources.

Coming Soon: Paged APIs

In the future the Rentpath API will use paging to conserve server resources and limit response size for resources that return potentially large collections of items.

Authentication

Each request to the Rentpath API will contain two parameters used for authentication. These parameters will be added to the querystring of every request.

  • access-key-id - the public access key id assigned to you by Rentpath
  • signature - the hmac signature you create for the request

The process for authentication

  1. You construct a request to the Rentpath API.
  2. You calculate an HMAC-SHA signature using your secret access key (assigned to you by Rentpath for api access). For more information, see http://www.faqs.org/rfcs/rfc2104.html.
  3. You include the access-key-id and signature parameters in your request uri.

Testing

For testing, simply include access-key-id and signature parameters with values of "test". The sample above including authentication parameters for testing: 
    https://api.reviews.apartmentguide.com/v1/reviews?property-id=999&date=01012013&access-key-id=test&signature=test

Errors & Validation

If a request fails due to client error, the resource will return an HTTP response code in the 40x range. These can be broadly categorised into:

HTTP Code Description
400 (Bad Request) One or more of the required parameters or attributes:
  • were missing from the request;
  • incorrectly formatted; or
  • inappropriate in the given context.
401 (Unauthorized) Either:
  • Authentication is required but was not attempted.
  • Authentication was attempted but failed.
  • Authentication was successful but the authenticated user does not have the requisite permission for the resource.
See the individual resource documentation for details of required permissions.
403 (Forbidden) Actions are usually "forbidden" if they involve breaching the licensed user limit of the server, or degrading the authenticated user's permission level. See the individual resource documentation for more details.
404 (Not Found) The entity you are attempting to access, or the project or repository containing it, does not exist.
405 (Method Not Allowed) The request HTTP method is not appropriate for the targeted resource. For example an HTTP GET to a resource that only accepts an HTTP POST will result in a 405.

For 40x HTTP codes the response will typically contain one or more validation error messages, for example: 

    {
        "errors": [
            {
                "context": "name",
                "message": "The name should be between 1 and 255 characters.",
                "exceptionName": null
            },
            {
                "context": "email",
                "message": "The email should be a valid email address.",
                "exceptionName": null
            }
        ]
    }

The context attribute indicates which parameter or request entity attribute failed validation. Note that the context may be null.

A 500 (Server Error) HTTP code indicates an incorrect resource url or an unexpected server error. Double-check the URL you are trying to access, then report the issue to a Rentpath API developer.
